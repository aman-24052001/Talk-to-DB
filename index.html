How it works

Question → guarded SQL → answer

01
You ask
Plain English. No SQL. Multi-turn conversation keeps context across questions.
02
Claude writes SQL
Live schema + 7 few-shot examples guide the agent to the correct join paths and formulas.
03
AST firewall
sqlglot parses every statement. Single SELECT? No forbidden nodes? Tables in schema? LIMIT enforced?
04
Read-only session
Even if something passes the AST, the DB session is physically read-only at the driver level.
05
You see everything
Answer, exact SQL, row count, guardrail stamps, agent turns, elapsed time. Full transparency.
06
SSE streaming
Real-time events per agent step — thinking, SQL attempt, firewall result — no silent waiting.

Security

The firewall catches everything harmful

Every statement the LLM produces is parsed to an AST by sqlglot before touching the DB. 10 attack vectors tested and blocked in CI.

🗑
DELETE
DELETE FROM orders
⊘ Only SELECT queries are allowed. Got: Delete.
💣
DROP TABLE
DROP TABLE t_shirts
⊘ Only SELECT queries are allowed. Got: Drop.
✏️
UPDATE
UPDATE t_shirts SET price=0
⊘ Only SELECT queries are allowed. Got: Update.
⛓
Multi-statement
SELECT 1; DROP TABLE orders
⊘ Exactly one statement allowed, got 2.
🔌
load_extension
SELECT load_extension('evil.so')
⊘ Function 'load_extension' blocked by policy.
👁
Schema exfil
SELECT * FROM sqlite_master
⊘ Table 'sqlite_master' not in allowed schema.
📋
SELECT INTO
SELECT * INTO stolen FROM customers
⊘ Forbidden operation in query: Into.
🔧
PRAGMA
PRAGMA table_info(t_shirts)
⊘ Only SELECT queries are allowed. Got: Pragma.
😴
pg_sleep
SELECT pg_sleep(30)
⊘ Function 'pg_sleep' blocked by policy.
Layer 2: Even if something slips the AST, the DB session is physically read-only at the driver level. INSERT/UPDATE/DELETE fail at the connection. 36/36 CI tests verify both layers hold on every push.
