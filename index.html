Multi-source

One question. Two databases. One answer.

Your sales sit in SQL. Your reviews live in MongoDB. You shouldn't have to know that. Ask one question — a planner decides which sources it needs, queries them in parallel, and a synthesizer writes you a single answer that draws on both.

How the hub stays honest
Each source is queried by its own independent agent, in its own thread, behind its own firewall — they don't even know they're part of a group. If one source is down, you still get an answer from the rest, with a clear note about what was missing. And the synthesizer only runs when more than one source actually returned data, so a single-source question costs nothing extra. Live progress streams in as it happens, every event tagged with which database it came from.
Security

The model proposes. The firewall disposes.

The language model is treated as an untrusted query author — for both SQL and Mongo. Every query it writes is checked in code before it touches a database, and the connection itself physically can't write. Here's what gets stopped, on each engine.

SQL firewall · sqlglot AST
🗑
DELETE / UPDATE
DELETE FROM orders
⊘ Only SELECT queries are allowed.
💣
DROP TABLE
DROP TABLE t_shirts
⊘ Only SELECT queries are allowed.
⛓
Multi-statement
SELECT 1; DROP TABLE orders
⊘ Exactly one statement allowed, got 2.
👁
Schema exfil
SELECT * FROM sqlite_master
⊘ Table not in allowed schema.
Mongo firewall · pipeline allowlist
📤
$out / $merge
[{ "$out": "stolen_copy" }]
⊘ Write stages are never allowed.
⚡
$where (arbitrary JS)
{ "$where": "function(){…}" }
⊘ Code-execution operators blocked.
🧬
$function, nested
$project → { "$function": … }
⊘ Caught at any nesting depth.
👁
Unlisted collection
find on users_pii
⊘ Collection not in allowed schema.
Layer 2 — the connection itself can't write. SQL sessions are opened read-only at the driver level (PRAGMA query_only, default_transaction_read_only). For Mongo, the connection object is capability-narrowed — insert / update / delete / drop methods aren't reachable on it at all, not merely permission-checked — plus a startup probe that warns if your credentials carry write rights. 111 CI tests verify both layers hold on every push, on both engines.
